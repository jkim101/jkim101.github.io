# GraphRAG vs. Standard RAG: 지식 그래프가 노이즈가 되는 순간 — 수학 교과서 검색에서 배우는 교훈

## Introduction: GraphRAG의 기대와 놀라운 반례

GraphRAG는 표준 RAG의 한계를 뛰어넘는 혁신으로 주목받아왔습니다. 숨겨진 관계를 연결하고, 다단계 추론을 가능하게 하며, 문서 전체에 대한 통합적 이해를 제공하죠. 그런데 최근 한 연구가 도발적인 질문을 던졌습니다.

"수학 교과서에 GraphRAG를 적용하면 어떻게 될까?"

개념 간 연결이 풍부한 도메인이니 GraphRAG의 강점이 빛을 발할 것 같지 않나요? 놀랍게도 결과는 정반대였습니다. 표준 RAG가 GraphRAG보다 페이지 단위 질의응답에서 더 우수한 성능을 보였습니다. 이것이 바로 '너무 많이 아는 것'의 역설입니다.

## Background: Standard RAG와 GraphRAG의 차이

표준 RAG는 단순합니다. 인덱싱 → 벡터 유사도 기반 검색 → 답변 생성으로 이어지는 직관적인 파이프라인이죠. 쿼리와 문서 청크 간 직접적인 텍스트 유사성에 집중합니다.

반면 GraphRAG는 한 걸음 더 나아갑니다. 엔티티와 관계로 구성된 지식 그래프를 구축하고, 단순히 매칭되는 텍스트뿐 아니라 문서 전체에 걸친 연결된 개념까지 검색합니다. 수학 교과서처럼 정리, 정의, 증명이 깊게 얽혀 있는 도메인이라면 GraphRAG에 완벽한 무대처럼 보였습니다.

## The Experiment: 수학 교과서 페이지 단위 QA 벤치마크

연구진은 'An Infinite Descent into Pure Mathematics'라는 순수수학 교재를 선택했습니다. 엄격한 논리 체계와 명확한 답이 존재하는 도메인이죠. GPT Vision OCR로 처리한 477개의 QA 샘플을 구축했고, 각 질문은 특정 교과서 페이지에 연결됩니다.

평가 기준은 두 가지였습니다:

- **검색 페이지 정확도**: 올바른 페이지를 찾았는가?
- **QA 유사도 F1 점수**: 생성된 답변이 정답과 얼마나 일치하는가?

핵심은 페이지 단위 정밀도를 요구한다는 점입니다. 주변 맥락에 흔들리지 않고 정확히 필요한 정보를 찾아낼 수 있는가를 테스트한 것이죠.

## Results: GraphRAG가 실패한 지점

GraphRAG는 검색 정확도에서 선전했습니다. Top-3 기준으로 최고 성능의 RAG 모델과 비슷한 수준이었죠. 관련 페이지를 찾는 능력은 충분했습니다.

하지만 답변 생성 F1 점수는 대부분의 표준 RAG 베이스라인보다 현저히 낮았습니다. 범인은 **중복과 과도한 콘텐츠**였습니다. GraphRAG는 타겟 페이지뿐 아니라 인접 페이지의 관련 엔티티까지 끌어와 답변 신호를 희석시켰습니다.

예를 들어 "피타고라스 정리가 어느 페이지에 있나요?"라는 질문에 GraphRAG는 피타고라스의 전기적 배경, 그의 제자들의 업적까지 끌어오며 정작 페이지 번호를 놓쳤습니다. 역설이 발생한 겁니다. GraphRAG의 강점(풍부한 맥락적 연결)이 정밀도가 요구되는 작업에선 오히려 노이즈가 된 것이죠.

## Root Cause Analysis: 입자도 불일치 문제

근본 원인은 GraphRAG의 작동 입자도(개념 수준, 페이지 간)와 작업이 요구하는 입자도(페이지 수준, 정밀 위치) 사이의 불일치입니다.

GraphRAG의 그래프는 추상적 개념을 연결합니다. 하지만 작업은 특정 콘텐츠가 담긴 물리적 페이지 위치를 필요로 했습니다. 표준 RAG의 벡터 유사도는 자연스럽게 검색 범위를 제한합니다. 가장 직접적으로 관련된 청크를 찾되, 주변부로 확장하지 않죠.

이것은 GraphRAG의 결함이 아닙니다. 도구 설계와 작업 요구사항 간의 정렬 실패입니다.

## Practical Guidelines: 언제 RAG를, 언제 GraphRAG를 사용할까

**GraphRAG를 사용하세요**:
- 전체 문서의 트렌드 요약
- 탐색적/추론 질문
- 분산된 정보를 연결해야 하는 다단계 쿼리

**Standard RAG를 사용하세요**:
- 특정 사실, 규정, 정의 검색
- 정밀도가 요구되는 작업
- 페이지/섹션 단위 위치 특정

핵심 원칙은 이것입니다: RAG와 GraphRAG는 계층 관계가 아닌 보완적 도구입니다. 선택은 질문 유형과 데이터 구조에 따라 결정되어야 합니다.

## Looking Forward: 정밀 작업을 위한 더 나은 그래프 설계

개선 방향은 명확합니다. **페이지 중심 그래프 설계**를 고려해야 합니다. 노드가 추상 개념이 아닌 페이지를 나타내고, 엣지는 구조적 관계(섹션 계층, 공식 의존성, 상호 참조)를 인코딩하는 방식이죠.

또한 제어된 확장 전략이 필요합니다. 초기 신뢰도가 낮을 때만 인접 노드로 검색을 넓혀 노이즈를 최소화하는 것입니다.

정확도/F1을 넘어서는 도메인별 평가 지표도 고려해야 합니다:
- 정의-공식 연결 품질
- 참조된 페이지 간 교육학적 일관성
- 레퍼런스 일관성

GraphRAG는 이제 '도메인 최적화' 시대로 진입하고 있습니다. 만능 그래프 구축은 목적 지향적, 도메인 인식 그래프 설계로 전환되어야 합니다.

## Conclusion: 과대광고보다 유연성

교훈은 GraphRAG가 실패했다는 게 아닙니다. 데이터 구조와 작업 요구사항을 고려하지 않은 채 기술을 무비판적으로 도입하면 최적이 아닌 결과를 낳는다는 것입니다.

실무자들이 던져야 할 더 깊은 질문은 "어느 게 더 나은가?"가 아니라 "왜 맞지 않았고, 어떻게 적응시킬 수 있는가?"입니다.

GraphRAG의 미래는 두 방향에 있습니다: 도메인 최적화된 그래프 아키텍처와 표준 RAG의 정밀성과 지식 그래프의 맥락적 풍부함을 결합한 하이브리드 접근법. 기술은 결국 문제에 맞춰 진화해야 합니다.
