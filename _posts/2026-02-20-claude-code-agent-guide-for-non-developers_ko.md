# AI에게 '이 일 처리해 줘' 한 마디면 끝나는 시대

우리는 ChatGPT나 Claude.ai 같은 대화형 인공지능을 이미 일상에서 사용하고 있습니다. 번역을 부탁하거나, 기획안 초안을 작성하게 하거나, 복잡한 개념을 설명해 달라고 요청합니다. 그런데 이들은 본질적으로 '조언자' 역할입니다. 질문에 답을 주고 텍스트를 생성해 주지만, 그 결과물을 복사해서 문서에 붙여 넣고 저장하고 정리하는 일은 여전히 사람의 몫입니다.

Claude Code는 이와 근본적으로 다릅니다. 파일을 직접 생성하고, 수정하고, 실행까지 합니다. 단순히 답변을 제공하는 것이 아니라 컴퓨터 안에서 실제로 일을 처리하는 '팀원'입니다. 영문 논문 PDF를 넣으면 한국어로 번역된 PDF를 만들어 주고, 유튜브 영상 링크를 주면 자막을 분석해 블로그 초안을 작성합니다. 콘텐츠 팀, 전략 기획 팀, 세일즈 팀 같은 가상 AI 팀을 만들어 운영할 수 있다는 뜻입니다.

핵심은 이 모든 것이 코딩 지식 없이도 가능하다는 점입니다. 필요한 건 '내 업무를 명확하게 언어화하는 능력'입니다. 내가 어떤 일을 어떤 순서로 하는지, 각 단계에서 무엇을 판단하는지를 설명할 수 있다면, AI에게 전달해 자동화할 수 있습니다. 이 글에서는 5단계 프로세스를 따라 나만의 AI 에이전트를 직접 설계하고 구축하는 전 과정을 안내하겠습니다.

## 1단계: 환경 준비 — VS Code + Claude Code 설치와 프로젝트 생성

Claude Code를 사용하려면 먼저 작업 환경을 세팅해야 합니다. 게임기(VS Code)와 게임팩(Claude Code 확장 프로그램)을 준비하는 과정이라고 생각하면 쉽습니다.

우선 Visual Studio Code, 줄여서 VS Code를 설치합니다. 마이크로소프트에서 만든 무료 코드 에디터인데, 개발자가 아니어도 사용할 수 있도록 설계되어 있습니다. VS Code 공식 웹사이트에서 운영체제에 맞는 버전을 다운받아 설치하면 됩니다.

VS Code를 실행한 후, 왼쪽 사이드바에서 확장 프로그램 아이콘을 클릭합니다. 검색창에 'Claude Code'를 입력하면 확장 프로그램이 나타납니다. 설치 버튼을 누르고, 설치가 완료되면 Claude 유료 계정으로 로그인합니다. Claude Code는 Claude Pro 또는 Claude Team 구독이 필요합니다.

로그인 후 중요한 설정이 하나 있습니다. 단축키 슬래시(/)를 눌러 명령어 창을 열고 'config'를 입력합니다. 설정 화면에서 'allow dangerously skip permissions'를 활성화해야 합니다. 이 옵션을 켜두면 Claude Code가 파일을 생성하거나 수정할 때마다 매번 허가를 요청하지 않고 작업을 진행합니다. 처음에는 조금 불안할 수 있지만, 에이전트가 원활하게 작동하려면 반드시 필요한 설정입니다.

터미널을 통해 설치하는 방법도 있습니다. Mac 사용자라면 터미널에 명령어 한 줄을 입력하면 바로 설치됩니다. Windows 사용자는 먼저 Git Bash를 설치한 후 동일한 명령어를 실행하면 됩니다. 터미널 설치 방식은 조금 더 기술적이지만, VS Code 없이도 Claude Code를 사용할 수 있다는 장점이 있습니다.

추가로 Python과 Node.js를 설치하는 것도 권장합니다. 이 두 프로그램은 엑셀 파일을 처리하거나 웹사이트를 생성하는 작업을 수행하는 엔진 역할을 합니다. Claude Code가 이런 작업을 실행할 때 필요한 도구들입니다. 각 공식 웹사이트에서 다운로드하고 기본 설정으로 설치하면 됩니다.

이제 프로젝트 폴더를 만들 차례입니다. 여기서 중요한 개념이 하나 있습니다. 프로젝트 폴더는 '팀 단위'가 아니라 '작업 단위'로 구성해야 합니다. 예를 들어 '콘텐츠 팀' 같은 큰 범위로 폴더를 만들면, 논문 번역 작업과 블로그 작성 작업이 섞여서 에이전트가 혼란스러워합니다. 대신 '논문 번역', '블로그 작성' 같은 구체적인 작업 단위로 폴더를 만들고, 각 작업마다 하나의 에이전트를 할당하는 것이 효과적입니다.

여러 작업을 동시에 진행하고 싶다면, VS Code 창을 여러 개 띄워 병렬로 작업할 수 있습니다. 각 창마다 다른 프로젝트 폴더를 열어두고, 각각의 에이전트가 독립적으로 작업하게 하는 방식입니다. 한 창에서는 논문을 번역하고, 다른 창에서는 블로그를 작성하는 식으로 동시에 여러 일을 처리할 수 있습니다.

## 2단계: 에이전트 설계서 작성 — 가장 중요한 단계

환경 설정이 끝났다면, 이제 에이전트에게 무슨 일을 어떻게 시킬지 정확히 설계해야 합니다. 이 단계가 전체 프로세스에서 가장 중요합니다. 에이전트 설계서가 명확하면 이후 단계는 거의 자동으로 흘러갑니다. 반대로 설계서가 모호하면 에이전트는 엉뚱한 결과물을 만들거나 중간에 길을 잃습니다.

에이전트 설계서에는 세 가지 핵심 요소가 담겨야 합니다.

**첫 번째는 What입니다.** 작업의 상세 절차를 단계별로 정리합니다. 예를 들어 논문 번역 에이전트라면, 'PDF 파일을 받아서 텍스트를 추출한다', '각 섹션을 번역한다', '그림과 표는 원본 그대로 유지한다', '번역된 텍스트와 이미지를 조합해 새 PDF를 생성한다' 같은 구체적인 단계를 나열합니다. 내가 직접 그 일을 할 때 어떤 순서로 하는지 떠올려 보면 됩니다.

**두 번째는 Why입니다.** 왜 이 작업을 하는지, 각 단계에서 무엇을 달성하려는지 의도를 설명합니다. 단순히 '번역하라'가 아니라 '학술적 맥락을 유지하면서 자연스러운 한국어로 번역해야 한다', '독자는 해당 분야 전문가이므로 전문 용어는 원어를 병기해야 한다' 같은 맥락을 제공합니다. 에이전트는 이 의도를 이해하고 판단 기준으로 활용합니다.

**세 번째는 How입니다.** 에이전트가 어떤 역할을 맡고, 어떤 도구나 API를 사용하며, 어떤 방식으로 작업을 수행할지 구체적인 방법을 정의합니다. 여기서 특히 중요한 것이 '에이전트(LLM)가 처리할 영역'과 '코드 스크립트가 처리할 영역'을 명확히 구분하는 것입니다.

번역, 판단, 비정형적인 텍스트 생성처럼 맥락을 이해하고 창의적으로 수행해야 하는 일은 에이전트의 몫입니다. 반면 파일 형식 변환, 반복적인 데이터 처리, 정확한 수치 계산처럼 정해진 규칙에 따라 정확하게 수행해야 하는 일은 코드 스크립트의 몫입니다. 이 구분이 명확하지 않으면 에이전트가 단순 반복 작업에서 실수하거나,
